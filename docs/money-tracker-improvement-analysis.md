# 머니트래커 개선 분석 (2026-04-11)

## 1) 현 구조에서 확인한 병목

### A. 데이터 로딩/복구 전략
- 클라이언트 훅(`useMoneyTrackerData`)은 4개 API를 병렬로 호출하지만, 실패 원인/부분 성공 여부를 사용자에게 구조적으로 노출하지 않습니다.
- 현재는 "전부 실패"일 때만 에러를 크게 보여주고, 부분 실패는 `-` 또는 빈 차트로 남기기 쉬운 구조입니다.

### B. 신뢰도(품질) 메타데이터 부족
- 각 지표에 대해 `source`, `isEstimated`, `asOf` 같은 품질 메타가 없어,
  - FRED 실측값인지
  - fallback 추정값인지
  - 몇 분/몇 시간 지연인지
  를 사용자가 구분하기 어렵습니다.

### C. 탭별 중복 계산/표시 로직
- 총량, 침투율, 추세 등 파생 지표가 탭별로 분산 계산됩니다.
- 동일 지표가 탭마다 다르게 보일 여지가 있고(정밀도/조건 분기), 유지보수 비용이 큽니다.

### D. 차트 렌더링 안정성
- 일부 차트는 데이터 길이 체크가 강화됐지만, 탭 전환 시 차트 생성/파기 로직이 분산되어 있습니다.
- 향후 차트가 늘면 메모리/렌더링 이슈 재발 가능성이 있습니다.

## 2) 우선순위 개선안

## P0 (즉시)
1. **데이터 품질 레이어 도입**
   - `MonetaryData`/`DefiStatsData`에 필드별 메타 추가:
     - `source: 'fred' | 'fallback' | 'hybrid'`
     - `isEstimated: boolean`
     - `asOf: string`
   - KPI/카드 우측 상단에 "실측/추정" 배지 노출.

2. **부분 실패 UX 표준화**
   - 값이 없을 때 `-` 대신 원인 배지 표시:
     - `데이터 지연`, `API 제한`, `추정치 대체`.
   - 상단 헤더에 "현재 데이터 완전성"(예: 14/18 지표) 표시.

3. **API 응답 스키마 고정**
   - `/api/monetary` 응답에 `diagnostics` 추가:
     - series별 성공/실패
     - 마지막 성공 시각
     - fallback 개입 여부

## P1 (단기)
4. **파생지표 중앙화**
   - `app/money-tracker/utils/derivedMetrics.ts` 신설.
   - 침투율, 글로벌 유동성, 추세 계산을 공통 함수로 통합.

5. **차트 공통 훅 도입**
   - `useSafeChart` 훅으로 create/destroy/empty-data placeholder 표준화.

6. **자동 새로고침 정책 개선**
   - 현재 고정 5분 주기 → API 상태 기반 백오프(성공 시 5분, 실패 시 10~15분).

## P2 (중기)
7. **관측 가능성(Observability) 추가**
   - 서버 로그에 series별 실패 카운트/원인 축적.
   - 임계치 초과 시 슬랙/텔레그램 경고.

8. **성능 최적화**
   - 무거운 차트 컴포넌트 동적 로딩(`next/dynamic`).
   - 비활성 탭 차트는 lazy mount.

## 4) 성공 기준 (KPI)
- `-` 표기 비율: 현재 대비 70% 감소
- 부분 실패 발생 시 사용자 원인 인지율: 100% (배지/툴팁)
- 탭 전환 후 차트 오류(콘솔 에러): 0건
- 동일 지표(글로벌 M2/침투율) 탭 간 불일치: 0건
